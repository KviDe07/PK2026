# Система учёта абитуриентов — ФАКТ МФТИ

Набор Python-скриптов для приёмной комиссии: объединяет данные об абитуриентах из
**1С** (выгрузки Excel/CSV) и **Битрикс24** (REST API), отслеживает изменения статусов
во времени и формирует **Excel-отчёты**.

Что умеет:
- **Объединение** — сводит записи об одном абитуриенте из 1С и Битрикса в одну карточку
  (каскадное сопоставление: ID → email/телефон → нечёткое ФИО + дата рождения).
- **Отслеживание статусов** — ведёт историю смен статуса, не «откатывает» этап назад,
  отмечает недостающие документы.
- **Отчёты** — Excel с листами Master / Изменения / Аналитика / Review.
- **Безопасность данных** — спорные слияния не происходят автоматически, а уходят в
  очередь ручной проверки (лист Review).

Источник правды — локальная база **SQLite** (`data/db/applicants.sqlite`). Excel
используется только как формат выгрузки.

## Установка

Требуется Python 3.9+.

```bash
cd admissions
python3 -m venv .venv
source .venv/bin/activate            # Windows: .venv\Scripts\activate
pip install -e .                     # рабочие зависимости
pip install pytest                   # для запуска тестов (по желанию)
```

Команда `admissions` и вызов `python -m admissions` после установки эквивалентны.

---

## Быстрый старт

```bash
# 1. Создать базу данных
admissions initdb

# 2. Загрузить выгрузку из 1С
admissions ingest --file data/input/vygruzka_1c.xlsx

# 3. Подтянуть данные из Битрикс24 (нужен .env, см. ниже)
admissions sync-bitrix

# 4. Сформировать отчёты в data/output/
admissions report
```

## Этап разведки (выполнить ПЕРВЫМ на реальных данных)

Ключ связки 1С↔Битрикс и точные названия полей зависят от ваших выгрузок, поэтому
до боевого запуска осмотрите реальные данные и подгоните конфиги.

```bash
# Колонки выгрузки 1С и сверка с config/mapping_1c.yaml
admissions inspect data/input/vygruzka_1c.xlsx

# Поля сущности Битрикса и пример записи (нужен .env)
admissions inspect --bitrix
```

По результатам отредактируйте:
- **`config/mapping_1c.yaml`** — правую часть (названия колонок Excel/CSV).
- **`config/mapping_bitrix.yaml`** — сущность (`lead`/`deal`/`contact`) и коды полей
  (особенно пользовательские `UF_CRM_*`).
- **`config/settings.yaml`** — каскад сопоставления (если есть общий ID — впишите его
  в `matching.cascade[0].fields`), список статусов и `status_mapping`, правила документов.

## Конфигурация

| Файл | Назначение |
|------|-----------|
| `config/settings.yaml` | стратегия сопоставления, статусы и их приведение, правила документов, приоритет источников, пути |
| `config/mapping_1c.yaml` | колонка выгрузки 1С → каноническое поле |
| `config/mapping_bitrix.yaml` | сущность и коды полей Битрикса → каноническое поле |
| `.env` | секреты (`BITRIX_WEBHOOK_URL`). Скопируйте из `.env.example`, **не коммитится** |

### Битрикс24

Нужен входящий вебхук. В `.env`:

```
BITRIX_WEBHOOK_URL=https://ваш-портал.bitrix24.ru/rest/1/ваш_токен/
```

Без портала можно работать офлайн, выгрузив записи в JSON:

```bash
admissions sync-bitrix --from-file data/input/bitrix_export.json
```

## Команды

| Команда | Описание |
|---------|----------|
| `initdb` | создать/проверить схему БД |
| `inspect <файл>` / `inspect --bitrix` | осмотреть выгрузку (этап разведки) |
| `ingest --file <путь>` | загрузить файл 1С (с сопоставлением и слиянием) |
| `sync-bitrix` | выгрузить данные из Битрикс24 |
| `report [--type ...]` | сформировать Excel-отчёты |

Полезные флаги:
- `--dry-run` (у `ingest`/`sync-bitrix`) — показать, что изменится, ничего не записывая.
- `--force` (у `ingest`) — повторно загрузить файл, даже если он уже загружался
  (по умолчанию повтор по хешу содержимого пропускается — это даёт идемпотентность).
- `--type master|changes|analytics|review|all` (у `report`) — какие листы включить.
- `-v` — подробный лог.

## Как работает слияние и статусы

1. **Сопоставление** (каскад из `settings.yaml`, первое уверенное совпадение):
   собственный ID источника → явный общий ID → email/телефон → нечёткое ФИО + дата рождения.
2. **Неоднозначность защищена**: совпадение сразу с несколькими карточками или
   «похоже, но ниже порога» → запись уходит в лист **Review**, а не сливается.
3. **Приоритет источников** (`source_priority`, по умолчанию 1С > Битрикс): первичный
   источник перезаписывает поля, вторичный — только заполняет пустые. Расхождения по
   ключевым полям фиксируются как конфликты в Review.
4. **Статусы** приводятся к каноническим (`status_mapping`) и идут по воронке
   (`statuses.order`). Этап не «откатывается» назад; терминальные статусы
   (`rejected`, `withdrawn`) имеют приоритет. Каждая смена пишется в историю.

## Отчёты (`data/output/report_<дата>.xlsx`)

- **Master** — все абитуриенты, объединённые поля, текущий статус (с цветовой заливкой),
  колонка недостающих документов.
- **Изменения** — история создания карточек и смен статуса.
- **Аналитика** — воронка по статусам, разрезы по направлению/форме/основе, комплектность документов.
- **Review** — спорные слияния и расхождения полей для ручной проверки.

## Тесты

```bash
pip install pytest
pytest
```

Покрыты: нормализация, каскад сопоставления (включая уход в review),
слияние, история статусов, идемпотентность, генерация отчётов.

Тестовые фикстуры можно пересоздать: `python tests/make_fixtures.py`.
