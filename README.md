# Система учёта абитуриентов — ФАКТ МФТИ (ПК-2026)

Python-утилита для приёмной комиссии: синхронизирует данные об абитуриентах из
**1С** (выгрузка `.xls`) в **Битрикс24** (REST API через входящий вебхук).

Утилита **stateless** — нет локальной БД. Источник правды — Битрикс. Каждый запуск
читает свежую выгрузку 1С и приводит контакты и сделки в актуальное состояние.

## Модель данных

- **Контакт = идентичность абитуриента** (создаётся вручную оператором или утилитой):
  ФИО, телефон, почта, **Уникальный код** (UF-поле). Уровень поступающего — в стандартном
  **«Тип контакта»** Битрикса (TYPE_ID: CLIENT=Абитуриенты, SUPPLIER=Магистры,
  PARTNER=Аспиранты); sync ставит его из `settings.yaml → contact_type_id`.
- **Сделка = данные заявления** из 1С: конкурсная группа, сумма баллов, приоритет,
  согласие, БВИ, основание и т.д. Названия полей совпадают с колонками выгрузки 1С.
- **Одна сделка на конкурсную группу.** Если абитуриент подал несколько заявлений
  (на факультете 2 группы) — на каждое заявление своя сделка, контакт общий.
- **Связка:** контакт ↔ сделка по `CONTACT_ID`; абитуриент 1С ↔ контакт по
  **Уникальному коду** (а если кода ещё нет — по Фамилии+Имени).

## Логика синхронизации (`sync`)

Идёт от выгрузки 1С. Для каждого абитуриента:

1. **Найти контакт:** по уникальному коду → иначе по Фамилии+Имени среди контактов
   без кода (это «заготовки» оператора) → иначе создать новый контакт из 1С.
   Тёзки без кода → **пропуск + отчёт** (ничего не пишем).
2. **Обеспечить сделки:** на каждую конкурсную группу — отдельная сделка в воронке
   бакалавриата (создаётся при отсутствии).
3. **Заполнить данные сделки** из 1С — только изменившиеся поля. Стадию воронки
   (`STAGE_ID`) и заголовок (`TITLE`), которыми управляет оператор, утилита **не трогает**.

По умолчанию — **dry-run** (показывает план без записи). Запись — с флагом `--apply`.

**Разбор сопоставления.** Каждый `sync` пишет Excel-отчёт в `data/output`
(`sync_razbor_*.xlsx`) с листами: **Тёзки** (несколько контактов с одним ФИО без кода —
пропущены), **Не создались** (контакт не записался из-за контроля дублей Битрикса),
**Выбывшие** (код есть на контакте, но абитуриент пропал из новой выгрузки), **Дубли**.
Проблемные кейсы не угадываются автоматически — их разбирает человек по этому отчёту.

## Установка

Требуется Python 3.9+.

```bash
cd admissions
python3 -m venv .venv
source .venv/bin/activate
pip install -e .
```

Скопируйте `.env.example` → `.env` и впишите входящий вебхук Битрикс24:

```
BITRIX_WEBHOOK_URL=https://<портал>.bitrix24.ru/rest/<id>/<токен>/
```

## Команды

```bash
# 1. Разовая настройка Битрикса (идемпотентно; без --apply — предпросмотр)
admissions setup-contact-fields --apply   # поле контакта: Уникальный код (тип — стандартный TYPE_ID)
admissions setup-deal-fields    --apply   # поля сделки: данные заявления из 1С
admissions setup-funnel --from 0 --to 8 --apply   # скопировать стадии воронки (при необходимости)

# 2. Разведка
admissions inspect data/input/<файл>.xls   # колонки и абитуриенты из выгрузки 1С
admissions inspect --bitrix --entity deal  # поля сущности Битрикса

# 3. Синхронизация
admissions sync                            # dry-run: что изменится (свежий файл из data/input)
admissions sync --apply                    # записать в Битрикс
admissions sync --applications data/input/000000028.XLS --apply

# 4. Выгрузка сделок воронки в Excel (контакт + данные + стадия + комментарии)
admissions export                          # data/output/deals_export_*.xlsx
```

## Стадии (автоматизация в sync)
- Заявление из 1С без предыдущего контакта → новая сделка на стадии **«Поступившие заявления»**.
- Был контакт оператора (пустая сделка) и пришло заявление → её заполняем и двигаем в **«Связались»** (только вперёд — ручной прогресс не откатывается).
- Стадии задаются по названию в `settings.yaml → stages`.

## Выгрузка (`export`)
Команда `export` собирает все сделки воронки бакалавриата в Excel: контакт
(ФИО/телефон/почта/код/тип), данные заявления (Б:), стадию и **комментарии** —
таймлайн-заметки контакта и комментарии по сделке. (Имя автора комментария
требует у вебхука права `user`; при CRM-only вебхуке показывается только дата и текст.)

## Ручное заведение контакта оператором

Оператор создаёт контакт, заполняя **только Фамилию и Имя**. Уникальный код и тип
подтянет утилита при следующем `sync` (сопоставит с 1С по ФИО и проставит код).
После этого контакт связывается надёжно по коду.

Опционально в Битриксе можно настроить **робота** «при создании контакта → создать
сделку» для мгновенного появления сделки. Это не обязательно: `sync` всё равно
создаёт недостающие сделки и не плодит дубли (проверяет существующие по контакту и
конкурсной группе). Доступность роботов зависит от тарифа Битрикса.

## Структура

```
src/admissions/
  cli.py                 — командная строка
  config.py              — загрузка config/*.yaml + вебхук из .env
  normalize.py           — нормализация ФИО/телефона/почты/галочек/баллов
  ingest_applications.py — парсер выгрузки 1С (.xls), дедуп по (код, группа)
  applicant.py           — модель абитуриента 1С
  bitrix_client.py       — REST-клиент Битрикса (пагинация, batch)
  bitrix_fields.py       — создание/резолв полей контакта и сделки (по XML_ID)
  bitrix_funnel.py       — копирование стадий воронок
  sync.py                — ядро синхронизации 1С → Битрикс
config/
  settings.yaml          — пути, ID воронки, тип поступающего
  mapping_1c.yaml        — соответствие колонок выгрузки 1С
  mapping_deal.yaml      — параметры сделки, операторские (защищённые) поля
  mapping_contact.yaml   — значения «Тип поступающего»
```

## Веб-приложение для сотрудников

```bash
admissions-web          # или: python -m admissions.webapp
```
Поднимает сайт на **http://localhost:8000** (waitress). Возможности:
- **Загрузка из 1С** — выбрать уровень + файл `.xls/.xlsx` → **Предпросмотр** (что изменится,
  без записи) → кнопка **«Применить»** (запись в Битрикс). Плюс отчёт-разбор.
- **Выгрузка из Битрикса** — кнопка «Выгрузить сделки в Excel» (скачивание файла из `data/output`).
- **Селектор уровня** — бакалавриат активен; магистратура/аспирантура — задел (включаются
  в `settings.yaml → levels`, когда заведём их воронки и поля в Битриксе).

Деплой на сервер: waitress слушает `0.0.0.0:8000`, либо через gunicorn:
`gunicorn admissions.webapp:app`. Вебхук Битрикса берётся из `.env` на сервере
(один экземпляр для всей команды).

**Авторизация (HTTP Basic).** В `.env` задайте `APP_PASSWORD` (и при желании
`APP_USERNAME`, по умолчанию `admin`) — тогда вход в приложение требует логин/пароль.
Если `APP_PASSWORD` пуст — приложение открыто (только для локальной разработки; на
сервере пароль обязателен). Приложение пишет в живой Битрикс, поэтому без пароля
публиковать его нельзя.

## Деплой в Docker

На сервере с Docker:
```bash
git clone git@github.com:KviDe07/PK2026.git && cd PK2026
cp .env.example .env && nano .env      # BITRIX_WEBHOOK_URL + APP_PASSWORD (обязательно!)
docker compose up -d --build           # соберёт образ и запустит
# приложение на http://<сервер>:8000
```
- Реальные данные (загрузки 1С, выгрузки Excel) хранятся на хосте в `./data` (том), в образ не попадают.
- Секреты — только через `.env` (в образ не копируются; см. `.dockerignore`).
- Обновление версии: `git pull && docker compose up -d --build`.
- Логи: `docker compose logs -f`.

**Публичный доступ.** Порт 8000 отдаёт HTTP без шифрования. Для доступа из интернета
поставьте перед контейнером nginx/Caddy с HTTPS (Let's Encrypt) и не забудьте `APP_PASSWORD`.
Для внутренней сети/VPN можно и напрямую по 8000 (с паролем).
